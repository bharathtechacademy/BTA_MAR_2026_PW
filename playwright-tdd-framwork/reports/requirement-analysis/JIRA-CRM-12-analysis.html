📖 Narrative & Acceptance Criteria
User Story Narrative

As an existing user
When I log in to my Creatio CRM account
I want to use my business email and password through a secure and well-designed form
So that I can access my account and see errors when I enter invalid credentials

Context

This issue is focused on implementing basic form-level validations for the login form in Creatio CRM, ensuring that user input is properly validated both on the client side and communicated clearly through error messages.

✓ Acceptance Criteria
  1. Email Format Validation: The Business Email field should be highlighted in red and show the error 'Invalid email format' when the user enters an invalid email format and domain names.
  2. Email Length Validation: The Business Email field should be highlighted in red and show the error 'Invalid email format' when the user enters an email with more than 400 characters.
  3. Password Length Validation: The Password field should be highlighted in red and show the error 'Invalid value. Value must contain up to 100 characters' when the user enters a password with more than 100 characters.
  4. Invalid Credentials Message: The error 'Invalid email or password' should be displayed at the bottom of the page when the user enters an invalid username/password and tries to log in.
🎯 Story Objective & Detailed Interpretation
High-Level Objective

Implement client-side and server-side validation for the login form to ensure data integrity, improve user experience through immediate feedback, and enhance security by preventing invalid authentication attempts.

Detailed Interpretation
  • Client-Side Validation: Real-time validation as users type to provide immediate visual feedback (red highlighting and error messages)
  • Input Boundaries: Enforce strict character limits (400 chars for email, 100 chars for password)
  • Email Validation: Validate email format including proper domain structure (RFC 5322 compliant)
  • User Experience: Display clear, actionable error messages that guide users to correct their input
  • Security Posture: Generic error message "Invalid email or password" for failed authentication attempts (prevents email enumeration attacks)
  • Form State Management: Error states persist until corrected, ensuring user awareness
💼 What We Are Trying to Achieve
Business Goals
  • Improve User Experience: Reduce friction in the login process by providing clear guidance on input requirements
  • Reduce Support Burden: Minimize support tickets related to login issues by preventing invalid submissions
  • Enhance Security: Implement defense-in-depth with both client and server validation
  • Ensure Data Quality: Guarantee that only valid email addresses and properly formatted credentials enter the system
Testing Goals
  • Validation Coverage: Ensure all validation rules are implemented correctly
  • Error Messaging: Verify error messages are accurate, helpful, and properly positioned
  • Cross-Browser Compatibility: Test on Edge, Chrome, and Firefox
  • Edge Cases: Cover boundary conditions (max lengths, special characters, unicode)
  • User Flow: Verify end-to-end login flow including error recovery
  • Security: Ensure no sensitive information is exposed in error messages

⚠️ Risks Involved
R1: Client-Side Validation Bypass
Risk: Attackers can bypass client-side validation using developer tools.
Mitigation: Implement mandatory server-side validation; never trust client-side checks alone.
R2: Email Enumeration Attack
Risk: Distinct error messages for "user not found" vs. "wrong password" can leak registered email addresses.
Mitigation: Use generic error "Invalid email or password" as specified in acceptance criteria.
R3: Performance Degradation
Risk: Real-time validation on every keystroke could cause UI lag or excessive server calls.
Mitigation: Implement debouncing; consider local validation with periodic server sync.
R4: Browser Compatibility Issues
Risk: HTML5 validation attributes may render differently across Edge, Chrome, Firefox.
Mitigation: Test on all three browsers; use CSS to standardize visual feedback.
R5: Accessibility Compliance
Risk: Error messages not properly announced to screen readers (WCAG 2.1 Level AA).
Mitigation: Use ARIA attributes (aria-live="assertive", aria-invalid="true").
R6: Boundary Testing Gaps
Risk: Off-by-one errors at 100/400 character limits not caught in manual testing.
Mitigation: Automate boundary tests; test exactly at limits (99, 100, 101 chars).
